What each file does
index.html — Complete rewrite. All false claims removed: no more "data stays in your browser", "no account needed", "works offline". Replaced with accurate cloud messaging. All links use /auth not /auth.html. Copyright updated to 2026. SEO meta tags added.
confirm.html — New page at /confirm. Shows after signup when email confirmation is required. All functions gracefully fall back to base64 DB storage if Storage upload fails, so existing data keeps working.
logs.html + gallery.html — Updated to call getImageUrl() for display instead of using img.data directly. New uploads use _previewUrl for instant display without waiting for a signed URL round-trip.

🔍 Full Audit — What Should Be Done Next
🔴 Critical (Fix These First)
1. Landing page (index.html) is outdated and inconsistent
The live site still references localStorage, says "No account needed", "data never leaves your browser" — all of which are now false. The app is fully Supabase-powered. The landing page copy, FAQ, and feature list are all wrong and will confuse or mislead real users. This needs a complete rewrite to match the actual product.
2. index.html links still use .html extension
All CTA buttons on the landing page point to /auth.html instead of /auth. Since vercel.json is now in place, these should be /auth.
3. No email confirmation flow
When a user signs up with email confirmation enabled (live mode), they get stuck — there's no "check your email" screen, no resend button, and no feedback after clicking the email link. This needs a dedicated confirmation page or at minimum a proper handling state.
4. No password reset page
resetPasswordForEmail sends a link that redirects back to /auth.html — but there's no form to actually enter the new password. Users who click the reset link land on the login page and have no way to complete the reset.
5. Images stored as base64 in the database
This is a ticking time bomb. One trade with 3 screenshots can easily be 2–3MB of text in the trade_images table. At scale this will make queries slow, hit Supabase row size limits, and rack up storage costs. The fix is migrating to Supabase Storage (the bucket infrastructure is already planned).

🟡 Important (Do Soon)
6. No mobile responsiveness
The trade log table has min-width:920px — on mobile it's a horizontal scroll nightmare. The journal tabs overflow. The settings grid breaks on small screens. With traders increasingly logging on phones after sessions, this is a real gap.
7. journal.html is 75KB — way too large
At 1,355 lines it's becoming unmaintainable. The export/import modal, delete modal, tag management, and PIN logic should be split into separate included scripts. This also blocks caching — the entire file re-downloads on every change.
8. No error boundaries or offline handling
If Supabase is unreachable, users see silent failures or broken states. There's no "you appear to be offline" message, no retry logic, and no graceful degradation. A simple network status check and retry on the Supabase helpers would fix this.
9. Rate limiting on the Stripe Edge Function
Currently any authenticated user can call create-checkout repeatedly with no throttle. A bad actor could spam Stripe API calls. Add a simple per-user rate limit in the function.
10. No webhook signature validation fallback
If STRIPE_WEBHOOK_SECRET is missing or wrong, the webhook silently fails. Add a startup check that logs a clear error instead of failing invisibly.

🟢 Growth & Scalability
11. Risk Calculator
The landing page advertises it but it doesn't exist in the app yet. This is a false promise to potential users. It's also one of the most-wanted features for prop firm traders.
12. Trade notes are plain text — no formatting
Traders want to write structured notes — entry reason, exit reason, what to do differently. Even basic markdown support (bold, bullet points) would dramatically improve the journaling quality.
13. No CSV export
JSON export is for backup/restore. Traders also want to open their data in Excel or Google Sheets. A CSV export of the trades table would make TradeZona interoperable with other tools they use.
14. No notification / reminder system
Nothing nudges users to log their trades. A simple browser notification permission request asking "Remind me to log today's trades at 6pm?" would dramatically improve retention.
15. No account deletion flow
The dashboard has a "Delete My Account" button that says "Contact support". This is not acceptable under GDPR/CCPA. Users have a legal right to delete their data. This needs a real implementation via a Supabase Edge Function that deletes the user's profile, journals, trades, and images then calls auth.admin.deleteUser.
16. Supabase Service Role Key rotation
It was shared in this conversation and should be rotated immediately in Supabase settings if not already done.
17. No analytics on your own usage
You have no idea how many people sign up, activate, drop off, or convert to Pro. Add Plausible or Posthog (both privacy-friendly, free tier available) to track signups, journal creations, and upgrade clicks.
18. SEO is minimal
The landing page has no <meta description>, no Open Graph tags, no structured data. When someone shares the link on Twitter/Discord it shows a blank preview. Fix with 10 lines of meta tags.

🔵 Future Architecture (When You Scale)
19. Move to a proper framework
The current vanilla HTML/JS multi-file structure works but doesn't scale past ~10 pages. As features grow, shared state management becomes painful. A migration to Next.js or SvelteKit would give you: shared layouts, proper routing, SSR for SEO, component reuse, and TypeScript safety.
20. Supabase Storage for images
Migrate trade_images.data (base64 blob) to Supabase Storage bucket with signed URLs. This reduces database load by ~90% for image-heavy accounts.
21. Edge caching for analytics
Analytics currently re-fetches all trades on every tab switch. For Pro users with thousands of trades this will get slow. Cache the analytics result in sessionStorage and invalidate only when a new trade is added.
22. Supabase Row Level Security audit
As you add more Edge Functions (account deletion, subscription management) the RLS policies need an audit to ensure service role is only used where strictly necessary.
